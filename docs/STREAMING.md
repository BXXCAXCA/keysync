# Streaming chat

KeySync AI uses one frontend-facing event contract for all provider streaming adapters.

## Frontend contract

The frontend calls `start_chat_stream_with_key` with a provider config, a temporarily unlocked credential, and a unified chat request. The command returns a `streamId` and then emits `chat-stream-event` events.

Event payload shape:

```ts
interface ChatStreamPayload {
  streamId: string;
  event:
    | { type: "start" }
    | { type: "delta"; text: string }
    | { type: "usage"; inputTokens?: number; outputTokens?: number }
    | { type: "done" }
    | { type: "error"; code: string; message: string };
}
```

The UI appends each `delta.text` to the active assistant message and switches the composer from **Send** to **Stop** while a stream is active.

## Supported provider paths

| Provider kind | Endpoint style | Parser |
| --- | --- | --- |
| `openai_chat` | `POST /chat/completions` with `stream: true` | `choices[0].delta.content` |
| `openai_compatible` / `custom` | `POST /chat/completions` with `stream: true` | `choices[0].delta.content` |
| `openai_responses` | `POST /responses` with `stream: true` | `response.output_text.delta` |
| `google_gemini` | `:streamGenerateContent?alt=sse` | `candidates[0].content.parts[].text` |
| `anthropic_claude` | `POST /messages` with `stream: true` | `content_block_delta.delta.text` |

## Stop behavior

`stop_chat_stream(streamId)` removes the stream ID from the active stream registry. The SSE read loop checks the registry before handling each chunk and emits `done` when it detects cancellation.

This is cooperative cancellation. It stops UI updates and exits the read loop, but it does not yet abort the underlying HTTP request at the transport level. A later improvement should store a cancellation handle per stream and abort the request immediately.

## Current limitations

- Only text deltas are emitted.
- Image attachments are represented in the unified request type but are not fully mapped for every provider yet.
- System prompt handling differs by provider because upstream APIs use different formats.
- Stream IDs are currently generated by the backend command; pre-generating them in the frontend would make strict event filtering race-free.
- Usage accounting is best-effort because providers expose usage metadata in different chunks.
